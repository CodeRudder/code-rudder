# Architecture Design Skill

## 概述

这个 skill 提供了系统化的架构设计工作流程，通过**多级索引**、**强制上下文注入**和**自动索引维护**，确保 AI 编程工具在开发过程中保持设计与实现的一致性，避免关键设计决策丢失。

## 核心特性

- **人类可读**：设计文档使用 Markdown 格式，易于阅读和编写
- **结构化元数据**：YAML front matter 便于 AI 解析和维护
- **三级索引**：项目 → 模块 → Feature，快速定位设计文档
- **强制上下文注入**：开发前必须阅读相关设计文档
- **自动索引维护**：AI 自动更新索引，保证一致性
- **设计追溯**：代码文件标注设计文档引用

## 工作流程

### 1. 开发前上下文注入（强制）
- 读取项目级索引 → 了解全局模块结构
- 读取模块级索引 → 了解模块内 features
- 读取 feature 索引 → 了解具体设计文档
- 读取设计文档 → 理解领域模型、API、流程
- 生成上下文摘要 → 确认后开始开发

### 2. 创建设计文档
- 领域模型文档（Markdown + YAML front matter）
- API 设计文档（基础信息 + 依赖关系）
- 处理流程文档（步骤 + 异常处理 + 跨模块协作）

### 3. 自动维护索引
- 创建文档时自动更新索引
- 修改文档时自动更新依赖关系
- 删除文档时检查引用并更新索引

### 4. 代码实现
- 代码文件头部必须标注设计引用
- 实现与设计保持一致

### 5. 一致性验证
- 检查文件存在性
- 检查双向引用
- 检查循环依赖
- 检查孤立节点

## 使用示例

### 触发 Skill

```
"设计用户注册功能"
"创建消息搜索的架构设计"
"规划权限管理模块的实现"
"设计这个API接口"
```

### 示例工作流

#### 1. 设计新 Feature

```
用户：设计用户注册功能

AI 工作流程：
1. 检查是否存在 user-management 模块
2. 创建 FEAT-USER-001 feature 索引
3. 创建 User 领域模型文档
4. 创建 register.md API 设计文档
5. 创建 user-registration.md 处理流程文档
6. 更新模块索引和项目索引
7. 生成上下文摘要供确认
```

#### 2. 开发 Feature

```
用户：实现 FEAT-USER-001

AI 工作流程：
1. 读取项目索引 → 了解全局结构
2. 读取 user-management 模块索引 → 了解模块内 features
3. 读取 FEAT-USER-001 索引 → 了解需要的设计文档
4. 读取 User.md, register.md, user-registration.md
5. 生成上下文摘要：
   - Feature: FEAT-USER-001 用户注册与登录
   - 核心功能：提供邮箱注册和登录功能，支持JWT认证
   - 领域模型：User（用户实体）
   - API：POST /api/auth/register
   - 流程：参数验证 → 邮箱检查 → 密码加密 → 创建用户 → 生成token
6. 开始实现代码，在文件头部标注设计引用
```

#### 3. 修改设计

```
用户：User 模型需要增加 phone 字段

AI 工作流程：
1. 读取 User.md
2. 修改 User.md，添加 phone 字段
3. 更新 User.md 的变更历史
4. 检查哪些 features 使用了 User 模型
5. 更新所有相关 feature 的索引（如果需要）
6. 提醒开发者：可能需要更新相关代码
```

## 文档组织方式

### 按模块组织

每个模块一个目录，内部包含：
- `features/` - Feature 索引文档
- `domain-models/` - 领域模型文档
- `apis/` - API 设计文档
- `process-flows/` - 处理流程文档

### 索引文件

**YAML 格式**（主要给 AI 解析）：
- 项目级索引：`project-index.yaml`
- 模块级索引：`modules/{module-id}/index.yaml`

**Markdown 格式**（人类可读）：
- Feature 级索引：`features/{feature-id}/index.md`

### 设计文档

**Markdown + YAML front matter**：
- 人类可读的正文内容
- AI 可解析的结构化元数据
- 支持表格、代码块、格式化

## 核心价值

### 解决"设计-开发断环"问题
- ✅ 强制开发前阅读设计文档
- ✅ 代码文件标注设计引用
- ✅ 设计变更自动通知影响范围

### 关键设计不丢失
- ✅ 共享领域模型明确记录被哪些 features 使用
- ✅ 跨模块依赖关系清晰
- ✅ 处理流程完整记录

### AI 编程工具有完整上下文
- ✅ 三级索引快速定位
- ✅ 设计文档详细描述
- ✅ 依赖关系清晰可见

### 验收和开发基于同一套设计
- ✅ 验收标准在设计文档中明确
- ✅ 测试要点在设计文档中记录
- ✅ 实现与设计保持一致

### 易于维护和追溯
- ✅ 变更历史记录
- ✅ 影响范围自动识别
- ✅ 索引一致性检查

## 设计文档内容

### 领域模型文档
- 字段定义（类型、约束、说明）
- 业务规则
- 使用场景
- 数据库映射
- 关联模型

### API 设计文档
- 基础信息（端点、方法、认证、限流）
- 请求定义（headers、body、参数）
- 响应定义（成功、错误场景）
- 依赖关系（调用的服务、使用的模型、被调用方）
- 副作用
- 测试要点
- 变更历史

### 处理流程文档
- 流程步骤（负责人、操作、验证条件、失败处理）
- 异常处理
- 跨模块协作
- 性能指标
- 测试场景

## 最佳实践

### 1. 设计优先
- 始终先创建设计文档
- 设计评审后再实现
- 实现中发现问题立即更新设计

### 2. 文档专注
- 一个领域模型一个文件
- 一个 API 端点一个文件
- 一个处理流程一个文件
- 通过引用关联文档

### 3. 变更可追溯
- 所有变更记录在变更历史
- 破坏性变更明确标注
- 提供迁移指南

### 4. 跨模块通信
- 明确记录跨模块依赖
- 异步通信使用事件队列
- 最小化同步跨模块调用

### 5. 共享资源管理
- 早期识别共享模型
- 记录哪些 features 使用
- 集中管理共享服务

## 注意事项

- ⚠️ 不能跳过上下文注入步骤
- ⚠️ 不能直接修改代码而不更新设计文档
- ⚠️ 不能创建设计文档而不更新索引
- ⚠️ 不能忘记在代码文件添加设计引用
- ⚠️ 不能重复定义模型（应该引用）
- ⚠️ 不能忽略跨模块依赖
- ⚠️ 不能忽略索引一致性检查警告

## 常见问题

### Q: 为什么要用三级索引？
A: 快速定位设计文档。项目级了解全局，模块级了解局部，Feature级了解细节。

### Q: 为什么设计文档用 Markdown 而不是 YAML？
A: Markdown 人类可读性好，支持表格和代码块，编辑体验更好。通过 YAML front matter 保留结构化元数据供 AI 解析。

### Q: 索引文件为什么用 YAML？
A: 索引文件主要是给 AI 解析的，不需要频繁人工阅读，YAML 更适合机器解析。

### Q: 如果忘记更新索引怎么办？
A: AI 会定期执行索引一致性检查，发现不一致会自动修复或提醒。

### Q: 设计文档太详细会不会太重？
A: 设计文档是渐进式的。简单 feature 可以简化，复杂 feature 可以详细。重点是保持一致性，而不是追求完美。

### Q: 如何处理设计变更？
A: 先更新设计文档，记录变更历史，识别影响范围，更新所有相关索引，然后修改代码。禁止反向操作。

## 技术决策

### 为什么强制上下文注入？
- ✅ 确保开发基于完整理解
- ✅ 避免遗漏关键设计
- ✅ 保持设计-实现一致性

### 为什么按模块组织而不是按文档类型？
- ✅ 模块内聚性高，相关文档集中
- ✅ 便于模块级别的变更管理
- ✅ 减少跨模块引用复杂度

### 为什么需要代码文件头部标注设计引用？
- ✅ 代码和设计双向追溯
- ✅ 新成员快速理解代码背景
- ✅ 修改代码时知道需要更新哪些设计
