# QA Acceptance Testing Skill

## 概述

这个 skill 提供了系统化的需求验收和 BUG 验证工作流程，帮助 AI coding 工具准确、完整、真实地按业务流程执行操作步骤，验证 API 结果和 UI 界面渲染结果。

## 核心特性

- **语言无关**：适用于任何技术栈（Node.js、Python、Java 等）
- **流程导向**：定义规则和工作流，而非具体实现
- **系统化验证**：确保完整和准确的验证
- **可追溯性**：每个需求都有唯一 ID 和清晰状态

## 工作流程

### 1. 生成需求清单
- 解析用户提供的需求（任何格式）
- 分配唯一 ID（REQ-模块-序号、BUG-模块-序号）
- 按模块分组
- 定义验收标准
- 识别依赖关系
- **保存位置**：`ai-docs/prds/{需求编号}-{模块}-{特性标题}-prd-list.md`

### 2. 生成测试步骤
- **简化版**（默认）：用于首次测试或简单需求
- **详细版**：用于重复失败或复杂需求
- **保存位置**：`ai-docs/qa-acceptances/{需求/bug编号}-{标题}-qa-acceptance.md`

### 3. 执行验证
- **API 验证**：请求、响应状态、响应结构、响应数据、副作用
- **UI 验证**：页面加载、元素存在、元素状态、元素内容、交互、导航、数据显示

### 4. 记录结果
- 更新需求清单状态（pending/in_progress/passed/failed/blocked）
- 记录失败详情（截图、日志、错误信息）
- 收集 Web 相关证据（HTML、CSS、Console、Network）

### 5. 生成报告
- 执行摘要（通过率、失败数）
- 按模块统计
- 失败需求详情
- 修复建议
- 多轮对比（显示改进趋势）
- **保存位置**：`ai-docs/qa-reports/{需求/bug编号}-{标题}-qa-report-round{验收轮数}.md`

## 使用示例

### 触发 Skill

```
"验收登录功能的需求"
"检查 BUG-123 是否修复"
"执行用户管理模块的验收测试"
"验证这个需求"
"测试这个 bug 修复"
```

### 需求 ID 命名规范

- 需求：`REQ-LOGIN-001`、`REQ-USER-002`
- Bug：`BUG-LOGIN-001`、`BUG-USER-002`
- 功能：`FEAT-PAYMENT-001`

### 优先级

- P0：关键（阻塞发布）
- P1：高（发布前必须修复）
- P2：中（应该修复）
- P3：低（可选）

### 测试方法

- `api`：API 端点测试
- `ui`：Web UI 交互测试
- `data`：数据库/存储验证
- `manual`：需要人工验证

## 多轮测试

每次测试迭代：
1. 递增 `test_round` 编号
2. 保留之前的测试结果
3. 重点关注之前失败/阻塞的项目
4. 生成对比报告显示改进趋势

## 最佳实践

1. **首次测试使用简化版测试步骤**，节省时间
2. **重复失败时切换到详细版**，精确定位问题
3. **收集完整证据**：截图、日志、HTML、CSS、Console、Network
4. **明确验收标准**：每个需求都有具体的通过条件
5. **追踪依赖关系**：确保按正确顺序测试
6. **生成可操作的报告**：帮助开发者快速定位和修复问题

## 注意事项

- 这个 skill 不包含具体的代码实现，而是提供规则和流程
- 适用于任何编程语言和技术栈
- 专注于 API 和 Web UI 测试
- 支持手动和自动化测试
